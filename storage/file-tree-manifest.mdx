Personal Notes uses a cached manifest system to avoid scanning S3 on every request. The manifest (`file-tree.json`) contains metadata about all markdown files and folders in your vault.

## Manifest Structure

The manifest is a JSON file stored at the root of your S3 bucket (or under your vault prefix). It follows a strict schema defined in `lib/file-tree-manifest.ts`.

### Schema Version

```typescript
export const FILE_TREE_MANIFEST_VERSION = 1;
export const FILE_TREE_MANIFEST_FILENAME = "file-tree.json";
```

### Node Types

The manifest contains two types of nodes:

<CodeGroup>

```typescript File Node
interface FileTreeFileNode {
  id: string;              // Relative path (e.g., "notes/readme.md")
  type: "file";
  name: string;            // Display name (e.g., "readme.md")
  path: string;            // Same as id
  parentId: string | null; // Parent folder id or null for root
  lastModified?: string;   // ISO 8601 timestamp from S3
  etag?: string;           // S3 ETag for cache validation
  size?: number;           // File size in bytes
}
```

```typescript Folder Node
interface FileTreeFolderNode {
  id: string;              // Relative path with trailing slash
  type: "folder";
  name: string;            // Display name
  path: string;            // Same as id
  parentId: string | null; // Parent folder id or null for root
  lastModified?: string;   // ISO 8601 timestamp
  childrenIds: string[];   // Sorted array of child node ids
}
```

</CodeGroup>

### Complete Manifest

```typescript
interface FileTreeManifest {
  metadata: {
    version: number;        // Schema version (currently 1)
    generatedAt: string;    // ISO 8601 generation timestamp
    checksum: string;       // MD5 hex checksum for validation
    nodeCount: number;      // Total number of nodes
  };
  nodes: FileTreeNode[];    // All files and folders
  rootIds: string[];        // Top-level node ids
}
```

## Building the Manifest

The manifest is generated by scanning your S3 bucket recursively.

<Steps>
  <Step title="Initialize scan">
    The builder starts at the bucket root (or vault prefix) and uses `ListObjectsV2Command` with delimiter `/` to list folders and files level by level.
    
    From `lib/file-tree-builder.ts:64`:
    ```typescript
    const response = await client.send(
      new ListObjectsV2Command({
        Bucket: bucket,
        Prefix: applyVaultPrefix(prefix),
        Delimiter: "/",
        ContinuationToken: continuationToken,
      })
    );
    ```
  </Step>
  
  <Step title="Process folders">
    Folders are identified from `CommonPrefixes` and added to a queue for recursive scanning:
    ```typescript
    const folderEntries = response.CommonPrefixes ?? [];
    for (const entry of folderEntries) {
      const folderId = ensureFolderPath(relative);
      folderQueue.push(folderId);
    }
    ```
  </Step>
  
  <Step title="Process files">
    Only markdown files (`.md` extension) are included in the manifest:
    ```typescript
    if (!relative.endsWith(".md")) {
      continue;
    }
    ```
    
    The manifest file itself is explicitly excluded:
    ```typescript
    if (relative === FILE_TREE_MANIFEST_FILENAME) {
      continue;
    }
    ```
  </Step>
  
  <Step title="Build hierarchy">
    Parent-child relationships are established using a `childMap`:
    ```typescript
    const parentId = getParentPath(relative);
    addChild(childMap, parentId, relative);
    ```
  </Step>
  
  <Step title="Generate metadata">
    A checksum is computed over the entire manifest for cache validation:
    ```typescript
    const checksum = createHash("md5")
      .update(checksumPayload)
      .digest("hex");
    ```
  </Step>
</Steps>

## CLI Commands

### Build Locally (Dry Run)

Generate the manifest without uploading:

```bash
pnpm tree:build
```

This writes the manifest to `.cache/file-tree/manifest.json` for inspection.

### Build and Upload

Generate and upload the manifest to S3:

```bash
pnpm tree:refresh
```

### Build, Upload, and Sync to Redis

Generate, upload to S3, and update the Redis cache:

```bash
pnpm tree:refresh -- --push-redis
```

<Warning>
  The `--push-redis` flag requires `UPSTASH_REDIS_REST_URL` and `UPSTASH_REDIS_REST_TOKEN` to be configured.
</Warning>

## Runtime Refresh

You can rebuild the manifest at runtime via the API:

```bash
POST /api/tree/refresh
```

This endpoint:
- Requires authentication (logged-in user)
- Optionally requires `TREE_REFRESH_SECRET` header for additional security
- Rebuilds the manifest from S3
- Uploads the new manifest
- Updates the Redis cache
- Revalidates Next.js cache tags

<Note>
  Set `TREE_REFRESH_SECRET` in your environment to secure the refresh endpoint. Without it, any authenticated user can trigger a rebuild.
</Note>

From `app/api/tree/refresh/route.ts:6`:

```typescript
const REFRESH_SECRET = process.env.TREE_REFRESH_SECRET;
```

## Manifest Validation

Every manifest loaded from S3 or Redis is validated against the schema.

From `lib/file-tree-manifest.ts:182`:

```typescript
export function validateFileTreeManifest(
  value: unknown
): FileTreeManifestValidationResult {
  // Validates:
  // - Metadata fields (version, generatedAt, checksum, nodeCount)
  // - Node structure (id, type, name, path, parentId)
  // - Node count matches metadata
  // - No duplicate node ids
  // - Folder nodes have childrenIds
  // - File nodes don't have childrenIds
}
```

Invalid manifests are rejected and the application falls back to S3.

## Storage Locations

The manifest is stored in three places:

<CodeGroup>

```text S3 (Source of Truth)
Bucket: your-bucket-name
Key: file-tree.json
# or with prefix:
Key: vault/file-tree.json
```

```text Redis (Cache)
Key: file-tree:manifest
Value: {
  body: string,        // JSON-stringified manifest
  metadata: object,    // Manifest metadata
  etag?: string,       // S3 ETag
  updatedAt: string    // ISO timestamp
}
```

```text Next.js Cache
Tag: file-tree-manifest
Lifetime: minutes
```

</CodeGroup>

## Cache Hierarchy

When loading the manifest, the system checks caches in order:

<Steps>
  <Step title="Next.js cache (memory)">
    Fastest. Returns immediately if valid.
  </Step>
  
  <Step title="Redis cache (network)">
    Fast. Returns if valid, otherwise falls through to S3.
  </Step>
  
  <Step title="S3 (source of truth)">
    Slowest. Always authoritative. Updates Redis on fetch.
  </Step>
</Steps>

From `lib/manifest-store.ts:126`:

```typescript
export async function loadLatestManifest(): Promise<ManifestRecord | null> {
  "use cache";
  
  cacheTag(MANIFEST_CACHE_TAG);
  cacheLife("minutes");
  
  // 1. Try Redis
  const fromRedis = await readManifestFromRedis();
  if (fromRedis) {
    return validateAndReturn(fromRedis);
  }
  
  // 2. Fallback to S3
  const fromS3 = await fetchManifestFromS3();
  
  // 3. Seed Redis
  await writeManifestToRedis(fromS3);
  
  return fromS3;
}
```

## Troubleshooting

### Manifest shows stale data

Revalidate the cache:

```bash
pnpm tree:refresh -- --push-redis
```

### Manifest validation fails

Check the build output for schema errors:

```bash
pnpm tree:build
```

Common issues:
- Duplicate node IDs
- Invalid timestamps
- Node count mismatch

### Redis cache not updating

Verify Redis credentials and manually inspect the cache:

```bash
curl "$UPSTASH_REDIS_REST_URL/file-tree:manifest" \
  -H "Authorization: Bearer $UPSTASH_REDIS_REST_TOKEN"
```
